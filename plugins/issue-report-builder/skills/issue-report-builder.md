# Issue Report Builder（研发反馈单生成器）

把用户随口描述的问题或需求，整理成研发团队**能直接动手**的反馈单。研发最怕的是「信息不全的反馈」——光说"这里有bug"，他们得反复追问操作步骤、参数、报错、环境，一来一回浪费几天。这个技能的核心价值就是**一次把信息收集齐**，让研发拿到文档就能复现、定位、修复。

## 工作流总览

```
1. 判定类型      → Bug 还是需求？（决定用哪个模板、收集哪些信息）
2. 收集已知信息   → 从用户描述里抽取，缺什么列清单
3. 主动采集证据   → 用 Chrome DevTools 复现、截图、抓控制台/网络（自动为主）
                  遇登录页先请用户登录，登录后继续自动采
4. 兜底补充      → 登录不了/内网/无权限等自动采不到时，明确告诉用户该手动提供什么
5. 把关质量      → Bug：信息是否够研发复现？需求：是否产品级通用需求？
6. 产出文件包     → <报告目录>/report.md + <报告目录>/assets/
```

不要跳过第 5 步。一份信息不全的 Bug 单、或一个换皮的定制需求，发出去等于没发——研发要么追问要么打回。把关是这个技能区别于"随便填个模板"的关键。

**一份单子可以包含多个条目**：用户一次测试常会发现好几个问题、或一次想提好几个需求。模板就是为此设计的——公共信息（环境、填写人、日期）在顶部只写一次，下面用「问题 1 / 问题 2…」或「需求 1 / 需求 2…」逐条描述。下文各步骤里说的"收集信息、采集证据、把关"，对**每一个条目**都要各做一遍。

## 第一步：判定类型 + 理清条目数

读用户的描述，判断是 **Bug（系统/使用问题）** 还是 **需求（功能改进）**：

- 出现了不符合预期的现象（报错、数据不对、点了没反应、样式错乱）→ **Bug**
- 希望系统"能做某事 / 做得更好"，当前没毛病只是不够用 → **需求**

判断不了就直接问用户一句。**一份单子只装一类**——Bug 和需求分两份文档，不混装（研发的处理流程不同）。如果用户这次既有 Bug 又有需求，就产出两份文档。

同时理清**这次有几个条目**：用户可能一口气描述了多个问题/需求。把它们一个个拆出来，每个条目后续都走完整流程；采集和把关时别把几个问题混成一团。

## 第二步：收集已知信息

先把用户已经给的信息抽干净，**再**列出还缺什么——不要一上来甩一长串问题清单，那会让用户烦。

对照对应模板（见下方）的字段，逐项核对：用户说清楚的就记下，没说的归入"待补充清单"。一次把待补充清单亮出来，而不是挤牙膏式一条条问。

关键上下文（研发最常追问、最容易漏的）：
- **环境**：哪个环境？（生产/测试/预发）哪个版本？什么浏览器？
- **账号/角色**：用什么账号、什么角色操作的？（权限相关 bug 常因此而起）
- **数据**：操作的是哪条数据？（订单号、用户ID 等关键标识，研发要拿它去查库）
- **复现频率**：每次都这样，还是偶发？

这些不是表单要你机械填空——它们是研发能否快速复现的命门。比如"偶发"和"必现"对研发的排查方向完全不同。

**环境是必问项，缺了就先问、别往下走**。只要用户没明确点名这次是哪个环境——没说"生产/测试/预发"，也没给出能推断环境的环境名/URL（如 `jc4a 测试环境`、具体域名）——就**主动要求用户提供环境名称**，再继续采集和写单。原因有两层：一是研发复现的第一件事就是"在哪个环境复现"，环境错了整份单子的证据都对不上号；二是后面要用 Chrome DevTools 自动访问，连去哪个地址都不知道就无从下手。别自己猜一个环境默认值填进去——猜错比留空更危险，研发会照着错环境查半天。问法可以直接：

> 这次问题是在哪个环境发现的？告诉我环境名称（如「jc4a 测试环境」），有访问地址也一并给我，我好去复现采证。

用户给了环境名/地址后，把它记入"上下文环境"，全单复用（个别条目环境不同的，在该条目内单独注明）。

## 第三步：主动采集技术证据（核心能力）

这是技能最省人力的部分。**只要页面能自动访问**，就用 Chrome DevTools 实际复现操作、采集证据，而不是干等用户截图。

采集流程（用 `chrome-devtools` 系列 MCP 工具）：

1. **导航到问题页面**（`navigate_page` / `new_page`）
2. **按用户描述的步骤复现操作**（`click` / `fill` / `fill_form` / `type_text`）——边操作边记录每一步，这些就是 Bug 单里的"复现步骤"
3. **截图**（`take_screenshot`）——在出问题的那一刻截图，保存到报告的 `assets/` 目录，文件名要能说明截的是什么（如 `step3-submit-error.png`）
4. **抓控制台报错**（`list_console_messages`）——研发最看重这个，JS 报错往往直接指向问题代码
5. **抓相关网络请求**（`list_network_requests` / `get_network_request`）——失败的接口请求（4xx/5xx）、请求参数、响应体，对定位前后端问题极有价值

采集时把每一步的"操作 → 现象"对应起来记录，这样复现步骤和截图能一一印证。

**根因分析点到为止**：基于采集到的请求/报错，可以给一句方向性的初步猜测（标注"供研发参考"），帮研发少走弯路。但做这份单子的是测试/反馈角色，不是研发——别陷进去逐行扒代码、翻源码、推断具体哪行写错了。你的价值是把客观证据采全，而不是替研发下结论；证据是硬的、猜测是软的，别把猜测写成定论。

详细的采集手法、截图命名规范、如何把网络请求整理进文档、根因分析的尺度，见 `references/evidence-collection.md`。

## 第四步：自动采集不到时，明确兜底

自动采集会遇到拦路虎：需要登录、内网环境、特殊权限、复杂前置数据。**遇到障碍不要硬试，也不要默默放弃**——明确告诉用户卡在哪、需要他提供什么。

**遇到跳转登录页：先请用户登录，而不是直接放弃自动采集**。访问页面时若被重定向到登录页、或提示未登录，说明只是缺登录态，页面本身大概率能自动采。这时**主动请用户完成登录**——别一看到登录页就退回手动兜底，那会白白浪费自动采集这个最省人力的能力：

> 我打开页面时被跳到了登录页，需要先登录才能继续自动采集。麻烦你在我这个浏览器窗口里完成登录（输账号密码 / 扫码都行），登录好了告诉我一声，我接着复现采证。
> 如果你这边登录不了（比如没有这个环境的账号、登录卡住、需要特殊权限），也直接告诉我，我转成手动方式带你采集，不耽误出单。

用户登录成功后，回到第三步继续自动采集（复现、截图、抓控制台/网络）。

**用户登录不了时，转纯手动兜底、照常出单**——不要卡在登录上反复重试。退回手动采集，把要用户做的事说清楚：

> 那我们走手动方式，这份单子照样能出齐。请你帮我：
> 1. 在出问题的那一刻截个图，存到 `<报告目录>/assets/` 下
> 2. 打开浏览器控制台（F12 → Console），把红色报错复制给我
> 3. （如果是接口问题）F12 → Network，找到失败的请求，把它的状态码、请求参数、响应内容复制给我

其他自动访问不了的情况（内网地址、特殊权限页面等）同样走这套手动兜底。把要用户做的事说具体到"按哪个键、点哪里、复制什么"，别让用户猜。用户贴来的截图，引导他放进 `assets/` 或告诉你路径，你来引用。

**后端日志**始终走手动：让用户从日志系统/服务器捞好相关时间段的日志贴给你，你整理进文档（必要时截取关键行并标注）。技能不直接连日志系统。后端日志是**每个问题内部**的一项（和"浏览器控制台错误"平级），跟着对应问题走，不是全局放一处——A 问题的日志放 A 问题下，B 问题的放 B 问题下。

## 第五步：质量把关

发出去之前，站在研发的角度审一遍。

### Bug 单：信息是否够复现？

逐条自检——研发拿到这份单子，能不能不追问就复现？
- 操作步骤是否完整到"照着点就能重现"？
- 输入参数、操作的数据标识是否给了？
- 截图能否看清问题现象？
- 控制台报错 / 失败接口是否附上了？
- 环境、账号、版本是否写清？

有缺口就回到第三/四步补，或明确告诉用户还差什么。

### 需求单：是否产品级通用需求？

这是你要主动把关的地方。研发明确要求"需求得是产品级通用需求，而非定制需求"——只服务一个客户/一个特殊场景的定制诉求，不该混进产品需求池。

发现需求偏定制时，**主动质疑，给出你的判断**，例如：

> 你这个需求"给 XX 客户的导出加一个他们专用的字段"，听起来是单点定制——只有这一个客户要。我的判断是它偏定制，不太适合作为产品通用需求提给研发。
>
> 不过如果背后其实是"用户希望导出字段可自定义配置"这样的通用诉求，那就值得做成产品能力。你看是哪种？

然后**由用户决定**：
- 用户认同偏定制、不坚持 → 丢弃，不出需求单
- 用户坚持要提 → 帮他**抽象成通用功能**：把"给A客户加X字段"重构成"导出字段支持用户自定义配置"，让需求站在产品层面，研发才好排期、才有复用价值

抽象的关键：剥离掉具体客户/场景的专有名词，找到背后那个**多数用户都会遇到**的一般化诉求。

多个需求时，**逐个把关**：可能有的通用、有的偏定制。偏定制且用户不坚持的就从单子里去掉，不要因为"顺手一起提了"就放进去稀释整单的质量。

## 第六步：产出文件包

在报告目录下直接产出 `report.md` 和 `assets/`，**不要再多套一层子文件夹**——多一层嵌套会让用户打包发送时路径变深、还容易让工具找不到文件。

报告目录的确定：用户指定了就用用户指定的；没指定就在当前工作目录下新建一个有意义的名字（如 `bug-报表导出乱码-20260610/`），然后**直接在它里面**放文件：

```
<报告目录>/
├── report.md          # 主文档，用下方对应模板
└── assets/            # 所有截图、附件，report.md 用相对路径 ./assets/xxx.png 引用
```

截图等附件**一律放 assets/，文档里用相对路径引用**——这样用户把整个文件夹打包发给研发，链接不会断。产出后告诉用户文件夹路径，提示"可以直接打包这个文件夹发给研发"。

## 模板

根据类型读取并填充对应模板：

- **Bug 单** → `assets/template-bug.md`
- **需求单** → `assets/template-feature.md`

模板是骨架不是镣铐：但**模板里要求的每一项，即使没有内容也要保留标题，并写明"无"/"无报错"/"不涉及"**——而不是把整节删掉或留空。原因：研发看到"后端日志：不涉及"，知道这一项你已经确认过、不是漏了；看到一片空白或干脆没这节，就会怀疑信息缺失、回来追问。占位是在替研发回答"这块到底有没有"。

**多条目填写要点**：
- 上下文环境（环境/版本/浏览器/账号、填写人/日期）填在顶部，只写一次；个别条目若环境不同，在该条目内单独注明。
- 每个条目用 `## 问题 N` / `## 需求 N` 起头，条目内的各项（问题现象、复现步骤、截图…）用 `###` 三级标题，之间用 `---` 分隔条目。标题里更新好总数（"共 N 个"）。
- 截图按条目区分命名，避免混淆：如 `q1-step2-error.png`、`q2-list.png`，文档里在对应条目下引用。
- 只有一个条目时，照样用这个结构，只是 N=1——不必为单条目另搞一套格式。
