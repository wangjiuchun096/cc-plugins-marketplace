# 证据采集手法（Chrome DevTools）

本文是 SKILL.md 第三步的展开。用 `chrome-devtools` 系列 MCP 工具实际复现问题并采集证据，目标是让研发"照着文档就能复现、看到报错就能定位"。

## 采集前：先确认环境，再确认能否自动访问

动手前先核对一件事：**这次是哪个环境、目标地址是什么**。用户没明确点名环境就先要环境名称（见 SKILL.md 第二步）——连去哪个地址都没定，`navigate_page` 无从谈起。

环境/地址确定后，用 `navigate_page` 打开目标页面：
- 能正常打开、且不需要登录态 → 走自动采集。
- **跳转到登录页 / 提示未登录** → 别直接退回手动。这通常只是缺登录态、页面本身能采。**先请用户在当前浏览器窗口完成登录**（见 SKILL.md 第四步的话术），登录好再继续自动采集。只有当用户明确表示登录不了（无账号、登录卡住、缺权限）时，才转手动兜底——不要在登录上反复硬试。
- 报无权限 / 内网地址打不开等其他障碍 → 直接走手动兜底（见 SKILL.md 第四步）。

## 复现操作并记录步骤

按用户描述的操作，一步步在页面上执行，**边做边把每一步记下来**——这些记录直接变成 Bug 单的"复现步骤"。

- 点击：`click`（先 `take_snapshot` 拿到元素 uid 再点）
- 填表单：`fill` / `fill_form` / `type_text`，**把填入的具体值记下来**，这就是"输入参数"
- 等待：操作后用 `wait_for` 等页面响应，避免截图截早了
- 每完成一个关键动作，记一行"操作 → 看到的现象"，方便后面和截图对应

要点：复现步骤要写到"换个人照着点也能重现"的程度，参数值用反引号标出具体内容。

## 截图

在**问题发生的那一刻**截图（`take_screenshot`），这是研发判断现象的第一手依据。

- 保存到报告的 `assets/` 目录下
- 文件名要自解释：`step3-export-error.png` 好过 `1.png`
- 多个问题节点就截多张，按发生顺序编号（`step1-`, `step2-`…）
- 文档里用相对路径引用：`![说明](./assets/step3-export-error.png)`，确保整包发送时链接不断
- 截全页还是截局部看情况——现象在视口外就截全页，现象明确就截到能看清问题即可

## 控制台报错

`list_console_messages` 抓控制台输出，**重点是 error 级别的红色报错**——JS 异常往往直接指向出问题的代码位置，是研发最看重的线索。

- 保留完整报错文本和堆栈，别只抄第一行
- 有多条时，挑与本次操作时间相关的；无关的旧报错可不收
- 整理进 Bug 单"浏览器控制台错误"一节，用代码块包裹保留格式

## 网络请求

`list_network_requests` 看请求列表，对失败的（4xx/5xx）或可疑的请求用 `get_network_request` 拿详情。这对区分"前端问题还是后端问题"极有价值。

收集失败接口时，记录这几样研发要用的信息：
- 请求方法 + URL
- 状态码（500 指向后端，400/403 多为参数或权限）
- 请求参数（payload / query）
- 响应体（后端返回的错误信息常在这里）

整理成 Bug 单"失败接口"小节。一个 500 + 后端返回的错误 message，常常比任何文字描述都更快帮研发锁定问题。

## 根因分析适可而止（重要）

采集到网络请求后，可以给出**方向性的初步判断**——比如"打开弹窗时没看到详情接口请求，疑似前端没拉详情数据"，这种线索能帮研发少走弯路，很有价值。

但要记住：**做这份单子的是测试/反馈角色，不是研发**。定位根因不是你的职责，点到为止即可。不要陷进去逐行扒代码、翻源码、推断具体哪个函数写错——那是研发的活，扒太深既越界又容易猜错反而误导。

把握尺度：给"现象 + 客观证据（请求/响应/报错）+ 一句方向性猜测，并标注'供研发参考'"。证据是硬的，猜测是软的，别把软猜测写得像定论。研发拿到客观证据自己能很快定位，你的价值在于把证据采全，而不是替他们下结论。

## 采集完整性自检

收尾前对照 Bug 单需要的证据过一遍：复现步骤、参数、截图、控制台报错、（如涉及接口）网络请求——齐了再产出文档；缺的能自动补就补，补不了就在兜底环节明确告知用户。
